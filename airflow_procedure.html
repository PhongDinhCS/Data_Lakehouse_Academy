🚀 Real Company Procedure: SMB → Airflow → Cloud

Step 1: Data Engineer Writes DAG

👨‍💻 Role: Data Engineer (DEV environment)

Engineer writes Python code to define the pipeline.

def extract(): read files from /mnt/mfgtrace/ def upload(): send files to ADLS extract >> upload
⬇️

Step 2: Save Code to Git

👨‍💻 Role: Data Engineer

Code is pushed to repository (version control).

git add . git commit -m "new airflow DAG" git push
⬇️

Step 3: CI/CD Deploys DAG

🤖 Role: Automation (GitHub Actions)

System automatically deploys DAG to Airflow server.

on push: copy DAG to airflow/dags/
⬇️

Step 4: Airflow Detects DAG

⚙️ Role: Airflow System

Airflow scans folder and shows DAG in UI.

Now you can:

  • Run DAG manually
  • Schedule daily job
  • Monitor logs
⬇️

Step 5: DAG Runs (Production Flow)

⚙️ Role: Airflow Scheduler

Every day at 6AM:

  • Read files from SMB (/mnt/mfgtrace/)
  • Upload to Cloud Storage
  • Log success
⬇️

Step 6: Data Stored in Cloud

☁️ Role: Storage System

Files saved like:

/adls/mfgtrace/log/2026/04/01/
⬇️

Step 7: Monitoring & Retry

👨‍💻 + ⚙️

If failure:

  • Airflow retries automatically
  • Engineer checks logs
⬇️

Step 8: Environment Flow

How code moves:

DEV → you test DAG UAT → business validates PROD → real data runs daily
⬇️

🔥 Final Big Picture

Engineer writes DAG → Push to Git → CI/CD deploys → Airflow runs daily → Data moves to Cloud → Monitor & improve
